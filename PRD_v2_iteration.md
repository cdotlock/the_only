# PRD: the_only v2 迭代 — 从架构到超级升级

> Version 2.1 | March 2026
> Status: Planning Phase

---

## 1. 现状总结

### 1.1 已完成的工作

**架构设计（Commit 1）**：
- 系统架构从单层平铺内存升级为三层结构（Episodic → Semantic → Core）
- 仪式管道从串行改为并行可并行阶段
- 引入“源智能图”做预排序，“知识存档”持久化交付的文章，“叙事弧”让每次交付成为有主题的故事线
- 完整的迁移方案、回滚策略和成功指标

**核心运行时（Commit 2）**：
- `memory_v2.py`：三层 JSON 内存的读写、校验、维护周期
- `narrative_arc.py`：根据候选条目构建叙事弧（Opening/Deep Dive/Surprise/Contrarian/Synthesis）
- `source_graph.py`：源画像（质量、可靠性、深度、偏差、新鲜度、独占性）与预排序
- `knowledge_archive.py`：文章索引、搜索、月度摘要、跨文章链接
- `migrate_v1_to_v2.py`：将 v1 的 `context.md` / `meta.md` 解析并迁移到三层 JSON
- `the_only_engine_v2.py`：兼容 v1 接口的交付引擎，增加预览、存档搜索、状态查看

**测试验证**：
- 所有 v2 自测通过（62+17+22+36+73+34 = 244 个测试）
- v1→v2 迁移脚本在真实数据上验证成功

### 1.2 待完成的工作

**集成层**：
- v2 运行时需要与 v1 脚本（`mesh_sync.py`, `the_only_engine.py`）集成
- 仪式管道需要更新以使用 v2 的运行时
- 触发机制（cron / OpenClaw skill）需要适配 v2

**性能优化**：
- JSON 文件 I/O 可能成为瓶颈（但目前数据量小，影响有限）
- 仪式管道中的并行执行需要优化（例如，源预排名和网络同步可以并行）

**功能增强**：
- 更智能的反馈收集（6 级评分系统）
- 更丰富的知识存档功能（跨文章链接、知识图谱）
- 更强大的协作合成功能（Kind 1118-1120）

## 2. 迭代目标

### 2.1 核心目标

**从“架构设计”到“超级升级”**：
- 将 v2 的架构设计落地为真正可用的系统
- 确保 v2 在 v1 的基础上有显著的提升（不是简单的重构）
- 使 v2 成为“第二代”的 the_only，而不仅仅是 v1 的改进版

### 2.2 具体目标

**1. 完整集成**：
- v2 运行时与 v1 脚本无缝集成
- 仪式管道完全使用 v2 的运行时
- 触发机制（cron / OpenClaw skill）完全适配 v2

**2. 性能提升**：
- 仪式执行时间减少 20%（通过并行执行和源预排名）
- 内存系统响应时间 < 1 秒（通过 JSON 优化）
- 知识存档查询时间 < 2 秒（通过索引优化）

**3. 功能增强**：
- 6 级反馈评分系统（0=忽略，1=打开，2=阅读，3=反应，4=讨论，5=行动）
- 跨文章链接和知识图谱
- 协作合成（Kind 1118-1120）
- 渐进式能力解锁（Progressive Capability Unlocking）
- 仪式预览模式（Ritual Preview Mode）
- 错误恢复和弹性（Error Recovery and Resilience）

**4. 用户体验**：
- 更直观的初始化流程（渐进式，不是一次性）
- 更丰富的社交命令（“show me your friends”, “find new agents”）
- 更好的错误处理和用户反馈

## 3. 功能矩阵：Keep / Improve / Kill / Add

### 3.1 KEEP（不变）

| 功能 | 原因 |
|------|------|
| Ruby 人格和声音 | 核心身份，已证明有效 |
| 信息层次结构（7 项原则） | 哲学支柱 |
| Nostr 网络协议 + secp256k1 身份 | 正确的架构，保持 100% |
| Echoes 系统（好奇心捕获 → 满足） | 最高满意度的功能 |
| 静默反馈循环（无调查） | 哲学上正确 |
| HTML 文章输出与叙事运动简报 | 美丽，差异化 |
| 每个文件一篇文章的命名约定 | 干净、独特、永久 URL |
| `the_only_engine.py` 多通道交付 | 工作正常，足够 |
| 基于画布的发布模型 | 简单、可靠 |

### 3.2 IMPROVE（改进）

| 功能 | v1 问题 | v2 目标 |
|------|---------|---------|
| **质量评分** | 5 个独立维度，无图分析 | 添加语义去重、叙事冗余检测、跨项目张力评分。引入“洞察密度”作为第 6 个维度，衡量每字的新信息量。 |
| **内存系统** | 扁平的 `context.md` + `meta.md` | 三层：情景（每次仪式）、语义（跨仪式模式）、核心（稳定身份/偏好）。结构化 JSON 模式用于机器可解析的状态。 |
| **信息收集** | 100 个项目广度下限 | 替换为“20 个项目深度下限”——调查的项目更少，但每个都经过彻底阅读和评估。引入源预排名以跳过低收益源。 |
| **合成** | 每个项目独立 | 仪式级叙事弧：开场/深入/惊喜/解决。强制在 ≥2 个项目之间进行交叉融合。引入“合成人格”——不同类型内容的不同声音。 |
| **初始化** | 前期加载 12 步 | 渐进式：第 1 天 = 仅 webhook + 搜索。网络、隧道、RSS 在第一周内根据 Ruby 的价值展示解锁。 |
| **反馈精度** | 4 级参与度评分（0–3） | 6 级，带行为指纹：0=忽略，1=打开，2=阅读（页面停留时间代理），3=反应，4=讨论，5=行动/分享 |
| **网络集成** | 广播文章 + 流言 | 协作合成、共享探索队列、联合问题调查、声誉加权内容合并 |
| **自我进化** | 固定周期长度（每 5/10 次仪式） | 自适应周期：基于信号密度的进化触发器，而不是仪式计数。强信号到达时快速通道。 |
| **上下文引擎** | 具有手动模式强制执行的 Markdown | JSON 支持，带模式验证。Markdown 渲染以提高可读性，但 JSON 是真实来源。 |
| **源健康跟踪** | `consecutive_empty` + `quality_avg` | 添加 `reliability_score`（正常运行时间 %）、`freshness_latency`（从发布到检测的时间）、`exclusivity_score`（此源提供其他源没有的项目的频率）。 |

### 3.3 KILL（删除）

| 功能 | 原因 |
|------|------|
| **100 项调查下限** | 鼓励广度扫描而不理解。替换为深度优先方法。 |
| **固定 3 轮搜索**（广度→深度→逆向） | 太死板。替换为适应性搜索，根据发现的内容进行响应。 |
| **`nano-banana-pro` 概念插图** | 外部依赖，不可靠，很少比文本中的精心选择的类比增加更多价值。从默认管道中删除；保留为可选增强。 |
| **画布清理（7 天到期）** | 用户丢失文章。替换为存档系统。 |
| **`meta.md` 60 行上限** | 强制过早信息丢失的任意约束。替换为具有每部分上限的结构化部分。 |

### 3.4 ADD（新增功能）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **仪式叙事弧** | 每个仪式都有一个主题（自动检测或用户播种）。项目被排序以讲述一个故事：开场背景 → 深入 → 惊喜 → 逆向 → 合成。 | P0 |
| **知识存档** | 所有交付内容的持久、可搜索存档。基于主题重叠的跨文章链接。每月“知识摘要”总结积累的见解。 | P0 |
| **自适应搜索策略** | 代替固定的 3 轮，搜索适应：初始广泛扫描 → 跟随最有希望的线索 → 如果线索耗尽则转向 → 仅当主导叙事出现时才进行逆向搜索。 | P0 |
| **深度优先评估** | 对于每个候选项目，在评分前阅读完整源内容（而不仅仅是标题/摘要）。根据实际内容质量而非元数据进行评分。 | P0 |
| **三层内存** | 情景（每次仪式印象）、语义（跨仪式模式）、核心（稳定身份）。每层具有不同的保留和压缩策略。 | P0 |
| **源智能图** | 关于源的结构化知识：覆盖图、偏差概况、质量轨迹、跨源冗余。用于在获取前对源进行预排序。 | P1 |
| **协作网络合成** | 当网络上的 2+ 个代理覆盖同一事件时，检测重叠并生成更丰富的联合合成。归功于所有贡献代理。 | P1 |
| **仪式预览模式** | 完整仪式的 `--dry-run`：收集、评分、合成，但输出到控制台而不是交付。用户可以在推送前批准、编辑或拒绝。 | P1 |
| **渐进式初始化** | 第 1 天：最小设置。第 2-7 天：Ruby 根据观察到的使用情况建议增强功能。到第 7 天，完全配置完成，用户从未感觉“设置”。 | P1 |
| **多人格配置文件** | 支持多个上下文配置文件（例如，“工作”、“个人”、“研究”），具有独立的获取策略、来源和交付渠道。通过命令切换。 | P2 |
| **阅读分析** | 跟踪阅读模式：打开哪些文章、估计阅读时间（通过滚动深度）、哪些被重新访问。反馈到评分中，无需调查。 | P2 |
| **网络辩论协议** | 当代理对文章的质量或解释存在分歧时，将分歧作为功能展示：“Ruby 说 X，但 Vesper 认为 Y。” | P2 |

## 4. 技术架构方向

### 4.1 集成层设计

**目标**：将 v2 运行时与 v1 脚本无缝集成，确保仪式管道完全使用 v2 的运行时。

**方案**：
1. **更新 SKILL.md**：将仪式管道从 A→F 更新为 0→5，并引用 v2 脚本。
2. **创建集成脚本**：创建一个 `ritual_runner.py` 脚本，作为仪式管道的入口点，协调 v2 脚本和 v1 脚本的调用。
3. **更新触发机制**：确保 cron 任务和 OpenClaw skill 调用新的 `ritual_runner.py`。

**集成点**：
- **Phase 0 (Pre-Flight)**：调用 `memory_v2.py --action read-all` 读取三层内存。
- **Phase 1 (Gather)**：调用 `source_graph.py --action rank` 进行源预排名，然后执行搜索。
- **Phase 2 (Evaluate)**：调用 `narrative_arc.py --action build` 构建叙事弧。
- **Phase 3 (Synthesize)**：执行合成（LLM 调用）。
- **Phase 4 (Output)**：生成 HTML，调用 `knowledge_archive.py --action append` 索引文章，调用 `the_only_engine_v2.py --action deliver` 交付。
- **Phase 5 (Reflect)**：调用 `memory_v2.py --action maintain` 运行维护周期，调用 `mesh_sync.py --action sync` 同步网络。

### 4.2 性能优化

**目标**：减少仪式执行时间 20%，提高系统响应速度。

**方案**：
1. **并行执行**：在 Phase 1 中，并行执行源预排名、网络同步和 Echo 队列检查。
2. **缓存**：缓存源智能图和知识存档索引，减少文件 I/O。
3. **异步处理**：将耗时的操作（如网络同步）移到后台任务。

### 4.3 功能增强

**目标**：实现 v2 PRD 中定义的所有 P0 和 P1 功能。

**方案**：
1. **6 级反馈评分系统**：更新 `feedback_loop.md` 和 `memory_v2.py` 以支持 6 级评分。
2. **跨文章链接和知识图谱**：增强 `knowledge_archive.py` 以支持跨文章链接和知识图谱查询。
3. **协作合成**：实现 Kind 1118-1120 事件，并更新 `mesh_sync.py` 以支持协作合成。
4. **渐进式能力解锁**：更新 `onboarding.md` 和 `SKILL.md` 以支持渐进式能力解锁。
5. **仪式预览模式**：更新 `the_only_engine_v2.py` 以支持仪式预览模式。
6. **错误恢复和弹性**：更新 `SKILL.md` 以支持错误恢复和弹性。

## 7. 风险和缓解措施

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 深度优先评估太慢 | 中 | 仪式需要 20+ 分钟 | 预排名源以跳过低收益源。设置收集阶段的硬超时（10 分钟）。 |
| 三层内存增加复杂度 | 中 | 状态管理中的更多错误 | 每次写入时进行模式验证。每次读取时进行完整性检查。 |
| 协作网络合成需要临界质量 | 高 | 功能在 <10 个活动代理时无用 | 为优雅降级而设计：单独仪式不受影响。协作合成是附加的。 |
| 渐进式初始化可能使功能未配置 | 低 | 用户错过可选增强功能 | Ruby 在检测到它们有帮助时主动建议增强功能。从不唠叨。 |
| JSON 内存文件失去人类可读性 | 中 | 调试更困难 | 生成 Markdown 投影与 JSON 一起供人类检查。JSON 是真实来源。 |

## 8. 待讨论的开放问题

1. **存档存储**：存档应该是本地的，还是代理应该可选地将他们的存档索引发布到网络以进行跨代理知识搜索？
2. **协作合成归因**：当两个代理对合并的合成做出贡献时，最终文章应该如何归功于每个贡献？
3. **内存层边界**：什么触发将信息从情景 → 语义 → 核心提升？纯频率，还是应该有置信度阈值？
4. **洞察密度评分**：如何在没有参考语料库的情况下衡量“每字新信息”？潜在方法：与用户的现有知识（来自语义记忆）进行比较并测量增量。
5. **渐进式初始化时机**：Ruby 在建议新功能时应该多积极？太被动 = 功能保持未配置。太积极 = 烦人。
